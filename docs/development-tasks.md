# 開発タスクリスト

このドキュメントは、[README.md](../README.md) の「[6. 開発計画（ロードマップ）](../README.md#6-開発計画ロードマップ)」を、開発チームへ発注しやすい粒度に分解したタスクリストです。  
開発チームは本書を起点にしつつ、要件・設計・エラー処理の詳細は各参照ドキュメントを確認してください。

---

## 1. 発注時の使い方

- **発注単位**：原則としてタスク ID 単位で依頼する
- **実装順**：Phase 1 → Phase 2 → Phase 3 の順で進める
- **成果物**：コードだけでなく、必要なテスト成果物・運用ドキュメントも納品対象に含める
- **レビュー観点**：AI/OCR の不確実性を前提に、人手確認の導線・監査性・再処理性を損なわないこと

## 3. 開発タスク一覧

### Phase 1：基盤整備（優先度：高）

#### P1-01 ユニットテスト整備

- **目的**：後段ロジックの品質を先に安定化し、OCR 結果の揺らぎに影響されにくいテスト基盤を作る
- **主な作業**
  - `ParseService` の日付・金額正規化ロジックのテスト追加
  - `ValidationService` の要確認判定ロジックのテスト追加
  - SUCCESS / NEEDS_REVIEW / ERROR の状態判定ロジックのテスト追加
  - 個人情報を含まないダミー OCR 結果をテストデータとして整備
- **参照**：README.md「6. 開発計画」、docs/design-speculation.md「2.4」「2.6」、README.md「8. テスト・レビュー成果物の管理」
- **完了条件**
  - 正常系と主要な異常系のテストケースが揃っている
  - テストデータに実名・実メールアドレス・実電話番号を含まない
  - テスト結果を `tests/` 配下で確認できる

#### P1-02 エラーハンドリング強化

- **目的**：失敗時の原因切り分けと再処理判断をしやすくする
- **主な作業**
  - 業務エラーと技術エラーの分類実装
  - ユーザー向け通知文を日本語短文で統一
  - ログシートへファイル名・エラー種別・通知文・対応方法を記録
  - 1件失敗でバッチ全体が停止しないようファイル単位で継続処理
- **参照**：README.md「5. エラー処理ポリシー」、docs/error-handling.md 全体、docs/design-speculation.md「2.9」
- **完了条件**
  - エラー分類が実装とドキュメントで一致している
  - エラー時の保存先・通知文・再処理可否が追跡できる
  - 想定主要エラーのログ出力内容をレビューできる

#### P1-03 冪等性・重複防止

- **目的**：同一証憑の二重登録を防ぎ、再投入時も安全に処理できるようにする
- **主な作業**
  - Drive ファイル ID を台帳またはログに保存
  - 同一ファイル ID の既存登録チェックを実装
  - 再投入時の挙動（スキップ / 要確認 / エラー）を明確化
- **参照**：README.md「6. 開発計画」、docs/error-handling.md「6. 再処理手順」
- **完了条件**
  - 同一ファイルの二重登録を防止できる
  - 重複検知時の記録方法と運用手順が定義されている

### Phase 2：設定・運用整備（優先度：中）

#### P2-01 設定管理の一元化

- **目的**：設定値の散在を防ぎ、運用引継ぎしやすい構成にする
- **主な作業**
  - フォルダ ID・シート ID・シート名・申請者名などを `CONFIG` に集約
  - 精算対象月の固定値方式 / フォルダ名方式の切替を実装
  - API キーはスクリプトプロパティ管理とし、コミット対象には含めない
  - サンプル設定値はプレースホルダー文字列で管理
- **参照**：README.md「6. 開発計画」、docs/design-speculation.md「2.1」「1.2」
- **完了条件**
  - 設定値の所在が一目で分かる
  - 実キー・実 ID をソースへコミットしない運用が守られている
  - 精算対象月の決定方式を切り替えられる

#### P2-02 運用ドキュメント整備

- **目的**：導入・月次運用・障害対応を担当者が自走できる状態にする
- **主な作業**
  - セットアップ手順書の作成
  - 月次運用手順書の作成（精算対象月の切替、フォルダ運用）
  - トラブルシューティングガイドの作成
- **参照**：README.md「4. フォルダ運用」「7. 開発・運用方針」、docs/error-handling.md「3〜6」
- **完了条件**
  - 新任担当者が手順書のみで初期設定・月次運用を実施できる
  - 障害時の一次切り分けフローが文書化されている

### Phase 3：品質・拡張（優先度：中〜低）

#### P3-01 再試行ポリシー

- **目的**：一時的な外部要因による失敗を自動回復しやすくする
- **主な作業**
  - API タイムアウト・一時障害時の再試行条件を定義
  - 再試行回数・待機時間・打ち切り条件を実装
  - 再試行結果をログに残す
- **参照**：README.md「6. 開発計画」、docs/error-handling.md「2. エラー通知文一覧」
- **完了条件**
  - 一時障害に対して過剰再試行せず回復を試みられる
  - 再試行の成否がログで確認できる

#### P3-02 通知連携

- **目的**：管理者が異常を早期に検知できるようにする
- **主な作業**
  - エラー発生時の管理者通知手段を決定
  - 通知条件（即時 / 集約 / 重要度別）を定義
  - 通知文面を運用しやすい形で整備
- **参照**：README.md「6. 開発計画」、docs/error-handling.md「4. 管理者対応フロー」
- **完了条件**
  - 管理者が重要障害を見落としにくい通知設計になっている
  - 通知条件と通知先がドキュメント化されている

#### P3-03 追加機能検討

- **目的**：基盤安定後の改善テーマを計画的に進める
- **主な作業**
  - 重複チェック精度向上の検討
  - 会計システム連携の要件整理
  - 拡張候補の優先順位付け
- **参照**：README.md「6. 開発計画」、docs/requirement.md「1.2 スコープ」
- **完了条件**
  - 現行スコープ外の追加開発候補が整理されている
  - 優先順位と前提条件が合意されている

## 4. 共通の受入条件

- README・要件定義書・設計仕様書・エラー処理詳細と矛盾しないこと
- OCR の不確実性を前提に、要確認フラグ・要確認理由・ログの導線を維持すること
- テストデータやドキュメントに個人情報を含めないこと
- 設定例にはプレースホルダーを使い、実際の API キーや各種 ID をコミットしないこと
