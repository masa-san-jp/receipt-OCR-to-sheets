## 1. システムの目的

### 1.1 概要

本ツールは、**Google Drive モバイルアプリのカメラ機能で画像化した経費証憑（レシート、領収書など）を Google ドライブにアップロードするだけで、Gemini API による OCR と必要項目の自動抽出を行い、立替経費精算申請の下書きデータを Google スプレッドシートへ自動登録する**ことを目的とする。

### 1.2 スコープ

- **本ツールが担うこと**：経費証憑の OCR 処理・項目抽出・スプレッドシートへの下書き登録
- **本ツールが担わないこと**：経費精算の審査・承認・勘定科目の確定・会計システムへの連携

### 1.3 自動化の範囲

本ツールは**申請下書きの自動化**を目的とする。**完全自動精算ではない**。  
OCR 精度には限界があり、書式の多様性・画像品質・税率表記の有無などにより抽出結果が不完全になる場合がある。そのため、**人による確認・修正を前提とした設計**とし、申請者が最終的に内容を承認してから経費申請に使用することを想定する。

### 1.4 全体フロー

```
📱 Google Drive モバイルアプリで経費証憑を撮影
        ↓
📂 「01_経費証憑取込」フォルダに画像が保存される
        ↓
🤖 Google Apps Script が定期的に自動実行
        ↓
🧾 Gemini API が OCR と項目抽出（構造化 JSON で返却）
        ↓
📋 スプレッドシートに経費申請用データを 1 行追加
        ↓
⚠️  必須項目不足・金額不整合などは「要確認」フラグを付与
        ↓
📁 処理済み画像を「02_処理済み経費証憑」フォルダへ移動・リネーム
```

## 2. 対象書類

本ツールは以下の書類フォーマットを処理対象とする。書式が異なっても、共通の経費精算項目へ正規化する。

| 書類種別 | 説明 | 主な記載項目 |
|----------|------|------------|
| レシート | 小売・飲食店等の販売レシート。消費税内訳が印字されることが多い | 店舗名・利用日・品目・合計額・税額 |
| 領収書 | 宛名・但し書き・金額・発行日が明記された正式な領収書 | 宛名・但し書き・金額・発行日・収入印紙の有無 |
| 簡易領収書 | 宛名記載を省略した簡略形式の領収書 | 日付・金額・発行者名 |
| 利用明細等 | クレジットカード明細・交通系 IC カード明細・ネット購入の注文確認メールなど | 利用日・支払先・金額・カード番号末尾 |

### 2.1 書式正規化の考え方

各書類は記載形式が異なるが、本ツールでは以下の方針で共通項目へ正規化する。

- **抽出できた項目のみ保存**し、記載のない項目は空欄またはnullとする
- 書類種別を Gemini API で判定し、種別ごとのコンテキストで抽出精度を高める
- 書式の違いによる判定困難な場合は「要確認」フラグを立てる
- 正規化が困難な場合でも OCR 生テキストを保存し、人手確認の参考にする

## 3. 機能要件

### 3.1 Drive 取込フォルダの定期監視

- GAS のタイムトリガーで設定した間隔（推奨：10 ～ 60 分おき）に「取込フォルダ」内の未処理ファイルを検出する
- JPEG / PNG / HEIC / WebP などの主要な画像形式を対象とする
- 処理中は重複実行を防ぐため、処理中フラグまたは一時フォルダ移動で排他制御する

### 3.2 Gemini API による OCR と構造化抽出

- 画像を Gemini API（`gemini-2.5-flash-lite` 等）に送信し、OCR と項目抽出を行う
- 抽出結果は定義された JSON スキーマに従い返却する
- 不明な項目は推測せず、空欄または `null` とする
- 確信度が低い場合や必須項目が欠落する場合は `needs_review: true` を返す

### 3.3 書類種別の判定

- Gemini API が画像の内容から書類種別（レシート / 領収書 / 簡易領収書 / 利用明細 / 不明）を判定する
- 種別が確定できない場合は「不明」として要確認フラグを付与する

### 3.4 スプレッドシートへの 1 行追記

- 抽出した項目を [4. 抽出対象項目](#4-抽出対象項目) に定義した列順でスプレッドシートに追記する
- 処理日時・精算対象月は GAS 側で設定し、OCR 結果とは別途付与する
- 画像 URL は Drive 上のファイル URL を自動設定する

### 3.5 処理済みフォルダへの移動・リネーム

- 処理完了後、ファイルを「処理済みフォルダ」に移動する
- ファイル名は [設計仕様書](経費証憑OCRツール_設計仕様書.md) に定義した命名規則に従ってリネームする
- 要確認フラグが立っている場合はファイル名に `[要確認]` プレフィックスを付与する

### 3.6 HEIC を含む画像の処理

- iPhone でよく使われる HEIC 形式を含む主要画像フォーマットを処理できること
- HEIC ファイルを Gemini API に送信する際は適切なメディアタイプを指定する

### 3.7 要確認フラグの付与

- [3.2](#32-gemini-api-による-ocr-と構造化抽出) に加え、GAS 側のバリデーションでも要確認フラグを付与する
- 要確認フラグが立っている行は視覚的に区別できるよう、シートの色付け等を行う（任意）
- 要確認理由は別列に記録し、確認作業の効率化を図る

### 3.8 OCR テキストの保存

- Gemini API が抽出した OCR 生テキストをスプレッドシートの専用列に保存する
- 抽出項目に誤りがあった場合の手動修正やデバッグに活用する

## 4. 抽出対象項目

スプレッドシートに記録する項目を以下に定義する。

| # | 項目名 | 型 | 必須 / 任意 | 設定元 | 説明 |
|---|--------|----|------------|--------|------|
| 1 | 処理日時 | 日時 | 必須 | GAS | GAS が処理した日時（YYYY-MM-DD HH:mm:ss） |
| 2 | 精算対象月 | テキスト | 必須 | GAS/設定値 | 何月分として精算するか（YYYY-MM 形式）。詳細は [5. 精算対象月の考え方](#5-精算対象月の考え方) 参照 |
| 3 | 書類種別 | テキスト | 任意 | Gemini | レシート / 領収書 / 簡易領収書 / 利用明細 / 不明 |
| 4 | 利用日 | 日付 | 任意 | Gemini | 支払・利用が発生した日（YYYY-MM-DD） |
| 5 | 発行日 | 日付 | 任意 | Gemini | 領収書等の発行日（YYYY-MM-DD）。利用日と同日のことが多いが別管理 |
| 6 | 支払先名 | テキスト | 任意 | Gemini | 店舗名・会社名など |
| 7 | 宛名 | テキスト | 任意 | Gemini | 領収書の宛名（「上様」含む） |
| 8 | 摘要 / 但し書き | テキスト | 任意 | Gemini | 但し書き・品目・サービス内容など |
| 9 | 合計金額（税込） | 数値 | 任意 | Gemini | 税込合計金額（円） |
| 10 | 10%対象額 | 数値 | 任意 | Gemini | 消費税率 10% 適用の課税対象額（円） |
| 11 | 10%税額 | 数値 | 任意 | Gemini | 消費税率 10% の税額（円） |
| 12 | 8%対象額 | 数値 | 任意 | Gemini | 消費税率 8% 適用の課税対象額（円）（軽減税率） |
| 13 | 8%税額 | 数値 | 任意 | Gemini | 消費税率 8% の税額（円）（軽減税率） |
| 14 | 非課税額 | 数値 | 任意 | Gemini | 非課税取引の金額（円） |
| 15 | 税率不明額 | 数値 | 任意 | Gemini | 税率が判別できない金額（円） |
| 16 | 主税率区分 | テキスト | 任意 | Gemini | 10% / 8% / 混在 / 非課税 / 不明 |
| 17 | 支払方法 | テキスト | 任意 | Gemini | 現金 / クレジットカード / 電子マネー / QR 決済 / 不明 など |
| 18 | 書類番号 | テキスト | 任意 | Gemini | 領収書番号・伝票番号など |
| 19 | 適格請求書発行事業者登録番号 | テキスト | 任意 | Gemini | インボイス登録番号（T + 13 桁） |
| 20 | 住所 | テキスト | 任意 | Gemini | 発行者の住所 |
| 21 | 電話番号 | テキスト | 任意 | Gemini | 発行者の電話番号 |
| 22 | 申請者 | テキスト | 任意 | GAS/設定値 | 経費申請者の氏名またはメールアドレス |
| 23 | 画像 URL | URL | 必須 | GAS | Google Drive 上の画像ファイルへの URL |
| 24 | OCR 抽出テキスト | テキスト | 任意 | Gemini | Gemini が認識した全テキスト（デバッグ・確認用） |
| 25 | 要確認フラグ | 真偽値 | 必須 | GAS/Gemini | TRUE / FALSE。要確認な場合は TRUE |
| 26 | 要確認理由 | テキスト | 任意 | GAS/Gemini | 要確認と判断した理由（複数ある場合は改行区切り） |
| 27 | 備考 | テキスト | 任意 | 手動 | 申請者が自由に入力できるメモ欄 |

> **注意**：「必須」は業務上の観点での必須度を示す。実際の抽出は証憑の記載内容に依存するため、必須項目でも抽出できない場合がある。その場合は要確認フラグを立てる。

## 5. 精算対象月の考え方

### 5.1 利用日との分離管理

本ツールは、レシートや領収書に記載された「利用日」とは別に、**精算対象月**を独立した項目として管理する。

立替経費精算は月次処理であることが多く、以下のような場合に利用日と精算対象月が一致しないことがある。

- 月末に利用した経費を翌月の精算で申請する
- 申請が遅延し、実際の利用月より後の月次締めで処理する
- 会社の締め日ルールにより、対象月の割り当てが利用日に基づかない場合がある

そのため、精算対象月は OCR 結果から推定するのではなく、**運用情報（設定値またはフォルダ情報）に基づいて決定する**。

### 5.2 精算対象月の形式

- 形式：`YYYY-MM`（例: `2026-03`）
- 精算対象月と利用日が同じ月であることもあるが、意味的に異なるため別列で管理する

### 5.3 精算対象月の決定方式

以下の2方式を用意し、運用環境に合わせて選択する。

#### 方式A：CONFIG 固定値方式（推奨シンプル版）

GAS スクリプトの `CONFIG` オブジェクトに精算対象月を固定値として設定する。

```javascript
CONFIG = {
  EXPENSE_MONTH: '2026-03'  // 処理する月次に合わせて手動更新
}
```

- **メリット**：実装が簡単、月次処理の切り替えが明確
- **デメリット**：毎月 CONFIG を更新する運用手順が必要

#### 方式B：取込フォルダ名方式（推奨実運用版）

月別の取込フォルダを作成し、フォルダ名から精算対象月を自動取得する。

```
01_経費証憑取込/
├── 2026-03/   ← このフォルダにアップロードされたファイルは精算対象月=2026-03
└── 2026-04/
```

- **メリット**：手動設定変更なしに月別管理が可能、フォルダで視覚的に管理できる
- **デメリット**：フォルダ命名規則の運用徹底が必要

## 6. 消費税率の仕分け

### 6.1 複数税率への対応

日本の消費税は現在 10%（標準税率）と 8%（軽減税率・食料品等）の複数税率が混在している。同一のレシートに複数税率が含まれることがあるため、本ツールは税率区分ごとの対象額・税額を独立した項目として保持する。

### 6.2 保持する税率項目

[4. 抽出対象項目](#4-抽出対象項目) に定義した以下の項目で仕分けを行う。

| 項目名 | 説明 |
|--------|------|
| 合計金額（税込） | 全税率を含む税込合計額 |
| 10%対象額 | 標準税率（10%）の課税対象額 |
| 10%税額 | 標準税率（10%）の消費税額 |
| 8%対象額 | 軽減税率（8%）の課税対象額 |
| 8%税額 | 軽減税率（8%）の消費税額 |
| 非課税額 | 非課税取引に該当する金額 |
| 税率不明額 | 税率が証憑から判別できない金額 |
| 主税率区分 | 証憑全体の主な税率区分（後述） |

### 6.3 主税率区分の値定義

| 値 | 意味 |
|----|------|
| `10%` | 全額が 10% 課税取引 |
| `8%` | 全額が 8% 軽減税率取引 |
| `混在` | 10% と 8% が混在している |
| `非課税` | 全額が非課税取引 |
| `不明` | 税率が証憑から判別できない |

### 6.4 判定方針

- レシートに税率表記（「10%」「8%（軽減）」「内消費税」等）が明示されている場合はその記載を優先する
- 税率内訳の記載がない場合は、無理に推測せず対応する税率欄を空欄とし、主税率区分を「不明」とする
- 合計金額と税額内訳の合計が一致しない場合は要確認フラグを付与する

### 6.5 判定の限界

以下の場合は判定が困難なため、要確認として人手確認を必須とする。

- 税率表記が存在しないレシート（小規模店舗・簡易レシートに多い）
- 軽減税率対象品目と標準税率品目が混在するが内訳が印字されていない場合
- インボイス未登録事業者発行の証憑で税額記載が不明確な場合

## 7. 非機能要件

### 7.1 セキュリティ

| 要件 | 内容 |
|------|------|
| API キー管理 | `GEMINI_API_KEY` は GAS のスクリプトプロパティに保存し、コード内に直接記述しない |
| アクセス制御 | Google Drive フォルダおよびスプレッドシートのアクセス権限は申請者・管理者のみに限定する |
| 機密情報の扱い | クレジットカード番号・口座番号など機密性の高い情報は抽出・保存対象から除外する |
| GAS 実行権限 | GAS スクリプトは必要最小限の OAuth スコープのみを要求する |

### 7.2 個人情報・機微情報への配慮

- レシート・領収書には個人の購買行動・移動情報・金融情報が含まれる場合がある
- スプレッドシートおよび処理済みフォルダへのアクセスは業務上必要なユーザーのみとする
- Gemini API に画像を送信する際は、Google の利用規約およびプライバシーポリシーに従う

### 7.3 可監査性

- OCR 生テキストをスプレッドシートに保存し、抽出結果の根拠を確認できるようにする
- 要確認理由を別列に保存し、どの項目に問題があったかを記録する
- 処理ログ（処理日時・ファイル名・処理結果）を別シートまたはログに記録することを推奨する

### 7.4 保守性・拡張性

- `CONFIG` オブジェクトに設定値を集約し、コードの変更なしに環境差異に対応できる構成とする
- 書類種別・抽出項目の拡張が容易な JSON スキーマ設計とする
- 将来的な機能追加（重複チェック・通知・会計連携等）を妨げないモジュール構成とする

### 7.5 パフォーマンス・信頼性

- GAS の最大実行時間（6 分）を考慮し、1 回のトリガー実行で処理できる件数の上限を実装時に考慮する
- エラーが発生した場合でも他のファイルの処理に影響しないよう、ファイル単位で try-catch を行う

## 8. 運用上の前提・制約

### 8.1 OCR 精度の限界

Gemini API による OCR は高精度だが、以下の条件で認識精度が低下する場合がある。

- 撮影時のピントずれ・手ぶれ・低照度
- 長尺・折れ曲がり・しわのあるレシート
- 手書きや装飾フォントの使用
- 文字が小さすぎる部分
- 光沢紙への印刷による反射

これらの場合は要確認フラグが付与されるが、フラグが付かない場合でも抽出精度に限界があることを利用者は認識すること。

### 8.2 人手確認の必要性

- 本ツールが出力するデータは**下書き**である
- 経費申請の最終判断は申請者が行い、スプレッドシートの内容を確認してから申請すること
- 要確認フラグが立っている行は必ず画像 URL から元の証憑を参照して内容を確認・修正すること

### 8.3 会計・税務判断の制限

- 税率区分・課税区分の判定は OCR 結果に基づく補助情報にすぎない
- 消費税の申告・控除の判断は税務・会計の専門家または会社規定に基づき行うこと
- インボイス（適格請求書）の要件充足判断は、人手で確認すること

### 8.4 書式差異への対応

- 各書類の書式差異は完全には吸収できない
- 特殊な書式・外国語証憑・印字品質の低い証憑は抽出精度が低下する場合がある
