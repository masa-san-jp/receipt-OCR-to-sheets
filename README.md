# 経費証憑 OCR ツール

Google Drive にアップロードした経費証憑（レシート・領収書など）を、Gemini API で OCR・項目抽出し、Google スプレッドシートへ自動登録するツールです。

## 1. プロジェクトの目的・概要

> **前身プロジェクト**: 本リポジトリは [google-drive-ocr-card](https://github.com/masa-san-jp/google-drive-ocr-card) を前身として、経費証憑（レシート・領収書）向けに再設計・発展させたものです。

本ツールは、経費精算の**申請下書き作成を自動化**することを目的とします。

- **完全自動精算ではありません**。申請者が最終的に内容を確認・承認して使用することを前提とした設計です。
- OCR の不確実性を前提とし、要確認フラグと要確認理由を残すことで**監査可能性**を確保します。
- Google Workspace（Drive / Apps Script / Sheets / Gemini API）に完結した構成で、外部サービスへの依存を最小化します。

---

## 2. 想定ユースケース

- 社員が Google Drive モバイルアプリで経費証憑を撮影・アップロードする
- GAS が定期実行し、自動で OCR・スプレッドシート登録を行う
- 申請者はスプレッドシートの内容を確認・修正し、月次の経費申請に使用する
- 処理できなかった証憑はエラーフォルダに格納され、担当者がフォローする

---

## 3. 処理フロー

```
📱 Google Drive に経費証憑をアップロード
        ↓
📂 01_取込 フォルダにファイルが保存される
        ↓
🤖 GAS タイムトリガーが定期実行
        ↓
🧾 Gemini API で OCR + 項目抽出（構造化 JSON）
        ↓
✅ バリデーション・要確認判定
        ↓
   | 状態 | Sheets 登録 | 移動先 | 備考 |
   |------|------------|-------|------|
   | SUCCESS | あり | `02_処理済み` | — |
   | NEEDS_REVIEW | あり（要確認フラグ付き） | `03_要確認` | 人の確認が必要 |
   | ERROR | なし | `04_エラー` | 担当者がフォロー |
```

- **SUCCESS**：必要項目が揃い、整合性も確認できた場合
- **NEEDS_REVIEW**：登録はできるが人の確認が必要な場合（項目の欠損・値の曖昧さなど）
- **ERROR**：処理継続が不可能またはデータ信頼性が確保できない場合

---

## 4. フォルダ運用

| フォルダ名 | 役割 |
|-----------|------|
| `01_取込` | 処理前の証憑を置く取込口。GAS はここを監視する |
| `02_処理済み` | 正常処理が完了した証憑の格納先 |
| `03_要確認` | スプレッドシートへ登録済みだが、内容の確認・修正が必要な証憑 |
| `04_エラー` | 処理が失敗した証憑の格納先。担当者がフォローする |

- フォルダの階層化は行わず、一覧性を優先します。
- `03_要確認` と `04_エラー` は意図的に分離します。要確認は「登録済みだが確認が必要」、エラーは「未登録で対応が必要」を意味します。

---

## 5. エラー処理ポリシー

- エラー通知文は**日本語の短文**で統一します（例：「日付読取不可、要確認」）。
- 処理に失敗した証憑は `04_エラー` フォルダに格納し、担当者がフォローします。
- 1 件の失敗がバッチ全体を止めない設計とします（ファイル単位の try-catch）。

エラー通知文の一覧・エラー分類・詳細な対応手順は以下を参照してください。

👉 [docs/error-handling.md](docs/error-handling.md)

## 7. 開発・運用方針

- **README は概要に留める**：詳細仕様・エラー対応・運用手順は `docs/` に集約します。
- **ドキュメントの責務分離**：同じ内容を複数箇所で管理しないよう、各 docs ファイルに責務を持たせます。
- **人によるフォローを前提とした設計**：自動化はあくまで下書き作成の支援であり、申請者が最終確認を行います。
- **AIの不確実性を前提とした設計**：OCR 結果を盲信せず、要確認フラグと要確認理由で人手確認の導線を残します。

### ドキュメント構成

| ファイル | 内容 |
|---------|------|
| `README.md` | プロジェクト概要・フロー・ロードマップ（本ファイル） |
| `docs/development-tasks.md` | 開発タスク詳細・発注単位の整理 |
| `docs/requirement.md` | 要件定義書（機能要件・非機能要件） |
| `docs/design-speculation.md` | 設計仕様書（アーキテクチャ・モジュール設計） |
| `docs/error-handling.md` | エラー処理詳細（通知文一覧・分類・対応手順） |
| `tests/` | テスト結果・テストスクリプト（後述） |
| `reviews/` | レビュードキュメント（後述） |

## 8. テスト・レビュー成果物の管理

### `tests/` フォルダ

テストに関連するファイルを格納します。

| 格納対象 | 説明 |
|---------|------|
| テストスクリプト | GAS ユニットテスト・結合テストのスクリプトファイル |
| テスト結果レポート | テスト実行ごとの結果サマリー（例：`test-result-YYYYMMDD.md`） |
| テスト用データ | ダミーの OCR 結果 JSON など、テストに用いるサンプルデータ（個人情報を含まないこと） |

> **注意**: テストデータに実名・実メールアドレス・実電話番号などの個人情報を含めないでください。

### `reviews/` フォルダ

コードレビュー・設計レビューの記録を格納します。

| 格納対象 | 説明 |
|---------|------|
| レビュードキュメント | PR レビューのサマリーや設計レビューの議事録（例：`review-YYYYMMDD.md`） |
| 指摘事項リスト | レビューで挙がった指摘事項と対応状況の記録 |
| 承認記録 | 設計・実装フェーズごとの承認エビデンス |
